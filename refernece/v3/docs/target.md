## 现况

经过 xlxz-wiki v2 的开发，已经基本上知道该如何实现我想要的功能

但是 v2 开发的时候过于的粗放和没有规范，很多功能都是一步一步 逐步添加，所以说里面会有很多：
1. 重复实现的部分
2. 没有规范的部分


请你创建一个经过精心设计过的架构，来规范 v2 中的代码，并实现我想要的功能


## 开发要求

### 开发规范

1. 先规划，后开发
   - 在开始开发之前，先规划好项目的结构和功能模块，确保代码的可维护性和可扩展性。
   - 使用 markdown 文档记录项目的设计思路和实现细节。
   - 在开发过程中，及时更新文档，确保文档与代码保持一致。

2. 代码规范
   - 使用 TypeScript 进行类型检查，确保代码的类型安全。
   - 使用 模块化 和 解耦 的方式组织代码，确保代码的可读性和可维护性。确保单个文件不要超过 400 行代码，否则ai 的处理难度将会增加。
   - 遵顼 高内聚，低耦合 的原则，确保每个单元的功能独立，职责明确。

3. 文件夹规范
    - `note/`：存放所有的文档内容，比如 本项目的开发目标、开发文档、开发进度，架构设计等。
    - `tests/`：存放测试用例的代码，使用 jest 来实现单元测试。
    - `docs/`：存放项目的文档，比如 API 文档、用户手册等。

4. 禁止任何的盲目猜测和想当然
    - 在开发过程中，禁止任何的盲目猜测和想当然，所有的代码都需要有明确的需求和设计。
    - 如果你不确定 vuepress 的某个 API 的用法或者某个功能的实现方式，请先查阅文档，我已经将所有的相关文档放在了 `reference/vuepress-doc/` 目录下。它同时也是 vuepress 的官方示例。

### 技术栈

使用 和 v2 相同的 技术栈。

使用 vue 而不是使用 ts 创建元素。
考虑引入 vue 的 响应式 能力，替换部分 ts 的逻辑。


### 迁移规范

在迁移和重构的过程中，必须确保以下几点：

1. **功能完整**：迁移后的代码必须保持原有功能的完整性。
2. **样式一致性**：迁移后的代码必须保持原有页面样式的一致性，包括布局、颜色、字体等。

### 注意事项

因为 roo code 默认会忽略 . 开头的所有文件和文件夹，所以说 .vuepress 目录下的所有文件和文件夹都会被忽略，所以说我创建了一个符号链接从 vuepress 文件夹 到 .vuepress 目录下。这样读写 vuepress 的文件和文件夹的时候，就可以直接使用 vuepress 文件夹。
